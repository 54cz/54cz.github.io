Android MVP

发表于 2017-06-30 | 阅读次数

暑假在家感觉时间过的好快啊

MVC模式

MVP是由MVC(Model View Controller)演变而来的，那就不得不提提MVC了。

在MVC模式中通常将XML文件视为View层，而Activity则视为Controller。而实际上Activity除了作为Controller对业务逻辑的处理还承担了View去负责视图显示。这样不但耦合度高而且Activity也变得特别臃肿，往后代码变得难读，不好维护，无法测试

当然如果就是写个简单应用Hello world 什么的自然就不需要模式框架了

MVP模式


这是google todo-mvp里的一张图，很好的体现了MVP三者之间的关系

  • Model 依然是实体模型和数据源
  • Presenter 处理业务逻辑完成View与Model间的交互
  • View 对应于Activity或者Fragment负责View的绘制以及与用户交互

MVP的核心思想

MVP 把 Activity 中的 UI 逻辑抽象成 View 接口，把业务逻辑抽象成 Presenter 接口，Model 类还是原来的 Model

这么一看MVP模式分离了Activity中的视图逻辑和业务逻辑抽象到View和Presenter的接口中去提高了代码的阅读性，降低耦合。

接口的作用类似给层与层之间制定的一种通信协议，两个不同的层级相互交流，只要遵守这些协议即可，并不需要知道具体的实现是怎样,将“请求”和“实现”分离开来，这就是接口的解耦

Activity也只用于展示UI，以及接收用户的输入，并承担一些生命周期的任务。不过这样也加大了开发的工作量，多了很多View和Presenter的接口，一开始不但写起来感到别扭也难以理清思路…那么MVP能带来些什么好处呢？

MVP模式的作用

  • Activity 代码变得更加简洁
    使用 MVP 之后，Activity 就能瘦身许多了，基本上只有 FindView、SetListener 以及 Init 的代码。其他的就是对 Presenter 的调用，还有对 View 接口的实现。这种情形下阅读代码就容易多了，而且你只要看 Presenter 的接口，就能明白这个模块都有哪些业务，很快就能定位到具体代码。Activity 变得容易看懂，容易维护，以后要调整业务、删减功能也就变得简单许多。

  • 避免 Activity 的内存泄露
    Android APP 发生 OOM 的最大原因就是出现内存泄露造成 APP 的内存不够用，而造成内存泄露的两大原因之一就是 Activity 泄露（Activity Leak）（另一个原因是 Bitmap 泄露（Bitmap Leak））。

    采用传统的 MVC 模式，一大堆异步任务和对 UI 的操作都放在 Activity 里面，比如你可能从网络下载一张图片，在下载成功的回调里把图片加载到 Activity 的 ImageView 里面，所以异步任务保留着对 Activity 的引用。这样一来，即使 Activity 已经被切换到后台（onDestroy 已经执行），这些异步任务仍然保留着对 Activity 实例的引用，所以系统就无法回收这个 Activity 实例了，结果就是 Activity Leak。Android 的组件中，Activity 对象往往是在堆（Java Heap）里占最多内存的，所以系统会优先回收 Activity 对象，如果有 Activity Leak，APP 很容易因为内存不够而 OOM。

采用 MVP 模式，只要在当前的 Activity 的 onDestroy 里，分离异步任务对 Activity 的引用，就能避免 Activity Leak。

还有其他的优缺点就不一一列举了，接下来才是重点

  • addedlittaks 用于创建或编辑任务
  • data 数据源
  • statistics：显示统计相关的任务
  • taskdetail：用于读取或删除一个任务
  • tasks：用于管理任务的列表（主界面）
  • util：帮助工具类
  • BasePresenter、BaseView：presenter和view的接口基类

Activity

activity在项目中是一个全局的控制者，通过FragmentManager的findFragmentById()方法得到activity中相应的实例，如果为null则创建fragment实例并通过ActivityUtils类中的addFragmentToActivity方法将fragment动态添加到activity。然后创建presenter实例，该构造方法有两个参数，一个是data包下的tasksRepository(代表M，下面会提到)，另个一是tasksView(代表V)，把实现了View接口的fragment实例作为presenter的构造函数参数传入并在构造函数中以setPresenter()把二者联系起来,这样presenter和view可以互相调用对方的方法了。这里可以发现M,V,P都凑齐了，P充当着指挥官的角色指挥着M和V。然后把下面是部分代码：

这里值得注意的的View接口中的isActive()的实现：

isAdded()方法如果返回true代表Fragment添加到了Activity，false代表没有添加。在异步任务中通过调用isActive()方法来给后台异步任务添加判断避免内存泄漏。通常我见到的做法是在Activity的onDestroy里将view的引用设为null，但是这么做难免会有些问题，如果Activity销毁了，那presenter里面持有的view就为null，那如果这时候耗时任务刚好执行完在调用view中的某个方法就可能会报空指针。

Model

项目中model层最大的特点是被赋予了数据获取的职责，与我们平常model层只定义实体对象截然不同，实例中，数据的获取、存储、数据状态变化都是model层的任务，presenter会根据需要调用该层的数据处理逻辑并在需要时将回调传入。这样model、presenter、view都只处理各自的任务，此种实现确实是单一职责最好的诠释。

总结

大体的思路我们就理清楚了，Activity负责创建Fragment（V）和Presenter（P），而Presenter（P）的创建需要用到Fragment（V）和Repository（M），Fragment（V）已经被Activity创建了，Repository（M）则通过依赖注入的形式创建，并且Repository（M）可以同时操作远程数据和本地数据，在Presenter（P）中将Presenter（P）传递给Fragment（V），这样就可以在事件响应时调用P中的方法进行业务逻辑处理。
